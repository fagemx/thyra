# Thyra 系統心智模型

> 整合 autoresearch 教訓 + Karvi 黑板架構 + Village Pack 單一入口
> 2026-03-13 定案

---

## 一句話

**人類用 YAML 設定意圖，Thyra 在固定裁判和邊界內自動決策，Karvi 執行，Edda 記憶。**

---

## 從 autoresearch 學到的三個前提

autoresearch 教的不是「讓 AI 跑研究」，而是**自治循環成立的條件**：

| 條件 | autoresearch 怎麼做 | Thyra 怎麼做 |
|------|---------------------|-------------|
| 固定裁判 | `evaluate_bpb()` 寫死在 `prepare.py`，AI 不能改 | `evaluateVillage()` 寫死在 code 裡，AI 不能改 |
| 可比較記錄 | `results.tsv` + `analysis.ipynb` | `DecideSnapshot` + `CycleMetrics` + `results view` |
| 人類設定集中 | 改 `program.md` 就好 | 改 `village.yaml` 就好 |

沒有這三個，自治循環只是「很忙」，不知道有沒有變好。

---

## 人類做三件事

人類的操作不是只有「寫 YAML」，而是三件事：

```
1. 設定    → 編輯 village.yaml（唯一設定入口）
              目標、規則、預算、風格、策略

2. 觀察    → 看 results view / audit / snapshots
              分數趨勢、keep/discard/crash 比例

3. 干預    → 批准或否決 medium/high 風險提案
              必要時 abort / supersede
```

**「單一入口」指的是設定入口，不是全部互動面。**
人類仍然需要看結果和做風險決策。

---

## Village Pack：設定怎麼變成系統可用的東西

人類寫一份 `village.yaml`，Village Pack compiler 算 diff，呼叫正確的 lifecycle 操作：

```
village.yaml 裡的...           compiler 做的事                   底層結果
───────────────────            ──────────────                    ────────
village: name, goal            VillageManager.create/update()    village row

constitution:                  ConstitutionStore.supersede()     constitution row
  rules, permissions, budget   （整體替換，不是 update）            + Karvi budget sync

chief:                         ChiefEngine.create/update()       chief row
  personality, constraints     （permissions 必須 ⊆ constitution）

laws:                          LawEngine.propose/revoke()        law rows
  category, strategy           （不能 update，只能 revoke + re-propose）

skills: [name list]            SkillRegistry.resolve()           skill bindings
                               （只綁定，不建立；必須是 verified）
```

**YAML 是 desired-state，compiler 算 diff。** Constitution 改了 = supersede。Law 改了 = revoke + propose。

## 固定裁判：evaluateVillage()

```
village.yaml           evaluateVillage()            village_score
  提供參數 ──→          公式寫死在 code ──→            每輪自動算
  （目標、預算、         AI 不能改
    門檻）              人類只提供參數
```

**不是** YAML → 任意公式。
**而是** YAML 提供參數 → 套進固定 evaluator → 產出分數。

評分器本體不可修改，這是「固定裁判」的核心意義。

---

## 記錄本：三層，不是一層

```
Layer 1: audit_log（事件帳本）
  誰做了什麼、什麼時候、compile 了什麼、dispatch 了什麼
  → 回答「發生了什麼」

Layer 2: DecideSnapshot + CycleMetrics（實驗記錄）
  每輪的 context、決策、分數、成本、rollback
  → 回答「哪個版本的 engine 更好」「keep / discard / crash 比例」
  → 等同 autoresearch 的 results.tsv

Layer 3: Edda（判例記憶）
  為什麼這次這樣做、過去類似情境結果
  → 回答「上次類似情況怎麼處理的」
  → 影響下一輪的 decide()
```

## Karvi 黑板：讓 Thyra 看得更快，不是替代 Thyra 決策

Karvi 廚房裡有一塊白板（`board.json`）：

```
board.json
├── taskPlan    → Thyra 用：看到任務進度
├── controls    ← Thyra 寫：同步 budget
├── signals     → Thyra 用：看到即時事件
├── insights    ✗ Thyra 不用（Karvi 自己的解讀）
└── lessons     ✗ Thyra 不用（Edda 負責判例）
```

**黑板提升 observe 品質，不取代 decide()。**

Thyra 跟 Karvi 的溝通：
- **命令**（寫）：REST API — dispatch、sync controls — 這沒問題
- **觀察**（讀）：目前靠 webhook → audit → 查 DB（太慢）→ 未來改成讀 board/SSE

**Thyra 不變成黑板。** 它是經理，不是白板。它讀白板上的資訊來做決策。

```
│   人類    │                       │  Thyra   │
│          │ ←── results view ──── │  (大腦)   │
│          │ ←── approval 請求 ─── │          │
└──────────┘                       └────┬─────┘
                                        │
                          ┌─────────────┼─────────────┐
                          │             │             │
                          ▼             ▼             ▼
                    ┌──────────┐  ┌──────────┐  ┌──────────┐
                    │  Karvi   │  │  Edda    │  │ evaluate │
                    │  (手腳)   │  │  (記憶)  │  │ Village  │
                    │          │  │          │  │  (裁判)   │
                    │ 執行任務   │  │ 判例查詢  │  │ 固定評分  │
                    │ 回報狀態   │  │ 決策記錄  │  │ 不可修改  │
                    └──────────┘  └──────────┘  └──────────┘
```

| 角色 | 職責 | 比喻 |
|------|------|------|
| **人類** | 設定意圖、觀察結果、干預風險 | 餐廳老闆 |
| **Village Pack** | 人類設定的唯一入口 | 點菜單 |
| **Thyra** | 在邊界內決策、調度、治理 | 餐廳經理 |
| **Karvi** | 執行任務、維護執行狀態（黑板） | 廚房 |
| **Edda** | 記憶判例、提供歷史參考 | 食譜筆記本 |
| **evaluateVillage()** | 固定評分，不可被 AI 修改 | 米其林評審員 |

## 一句話版本

**人類用 Village Pack 設定規則與目標，Compiler 把它翻成 Thyra 的治理 artefacts；Thyra 根據 Karvi 的執行狀態與 Edda 的判例記憶做 bounded decisions；固定裁判評分、記錄本累積結果；人類再根據結果調整下一版 Village Pack。**

## 你要記住的 5 句話

**1. Human 不是直接操作 Karvi。**
Human 主要操作的是 **Village Pack**，不是 task runtime。

**2. Village Pack 不是資料表。**
它是 **single authoring surface**，會被 compile 成多個治理物件。

**3. Thyra 是大腦，不是黑板。**
它負責治理、決策、風險與法律生命週期。

**4. Karvi 是執行黑板，不是治理大腦。**
它提供 task execution、progress、signals、board-like shared state。

**5. Edda 是判例記憶，不是即時狀態。**
它回答的是「以前類似情況怎麼做、結果如何」。

## 最短心智模型

```text
Human 寫規則
→ Village Pack 統一入口
→ Compiler 轉成治理 artefacts
→ Thyra 做 bounded decision
→ Karvi 執行
→ Edda 提供判例
→ evaluateVillage() 打分
→ Results/Audit/Snapshots 累積
→ Human 再調整
```
